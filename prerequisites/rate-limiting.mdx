---
title: "Rate limiting"
description: "How rate limits work and which response headers to use for backoff and retries."
---

API endpoints are rate-limited per client IP (and per phone number for payment creation) to protect against abuse. Two **tiers** apply different limits: a higher limit for general use (airtime, bills, messages, balance/status) and a lower limit for pay/payouts. When you hit a limit, the API returns **HTTP 429 Too Many Requests** and tells you when you can retry.

---

## Tiers

| Tier | Limit (per 60s window) | Endpoints |
|------|------------------------|-----------|
| **Standard** | **100** | Token, order (airtime, bills, etc.), balance, status, messages |
| **Payment** | **50** | Pay, payout, order with service=payout |

Counts are **per tier per IP**: e.g. 100 requests to token/airtime and 50 to payments in the same minute are allowed. Create also applies a separate limit per phone number (payment tier).

---

## Behaviour

- **Window:** Sliding 60-second window per key (IP or phone, per tier).
- **Limits:** See table above; configurable via environment variables on the server.
- **Scope:** Applied to token, create, order, pay, payout and related endpoints.

When the limit is exceeded, the server responds with **429** and a `Retry-After` header. All rate-limited responses (both success and 429) include standard rate-limit headers so clients can throttle proactively.

---

## Response headers

These headers are present on **every response** from rate-limited endpoints (2xx and 429).

| Header | Description |
|--------|-------------|
| **X-RateLimit-Limit** | Maximum number of requests allowed in the current window. |
| **X-RateLimit-Remaining** | Number of requests remaining in the current window (0 when at or over the limit). |
| **X-RateLimit-Reset** | Unix timestamp (seconds) when the current window resets. |

On **429** responses only:

| Header | Description |
|--------|-------------|
| **Retry-After** | Recommended wait time in seconds (e.g. `60`) before retrying. |

---

## Example: success response (standard tier)

```http
HTTP/1.1 200 OK
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 85
X-RateLimit-Reset: 1234567890
Content-Type: application/json

{"access_token":"...","issued_at":1234567890,"expires_in":3600}
```

## Example: payment tier

```http
HTTP/1.1 200 OK
X-RateLimit-Limit: 50
X-RateLimit-Remaining: 42
X-RateLimit-Reset: 1234567890
Content-Type: application/json

{"status":"ok",...}
```

---

## Example: rate limit exceeded (429)

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 60
X-RateLimit-Limit: 50
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1234567890
Content-Type: application/json

{"error":"Too many requests."}
```

---

## Best practices

- Use **X-RateLimit-Remaining** to throttle before hitting the limit.
- Use **X-RateLimit-Reset** to schedule the next request after the window resets.
- On **429**, wait at least **Retry-After** seconds before retrying.
- Implement exponential backoff when you receive repeated 429s.
