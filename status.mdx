---
title: "Status Codes"
description: "HTTP and transactional status codes returned by the Tupay API."
---

Tupay uses two layers of status codes:

- The **HTTP status code** indicates whether the request itself was valid.
- The **`status` field** in the JSON body describes where a transaction is in its lifecycle.

Use them together to decide whether to retry, show an error, or mark a payment as complete.

---

## HTTP status codes

| Code | Meaning |
|------|---------|
| `200` | OK |
| `400` | Bad Request |
| `401` | Unauthorized |
| `402` | Request Failed (e.g. insufficient balance) |
| `403` | Forbidden |
| `404` | Not Found |
| `409` | Conflict (e.g. duplicate request) |
| `429` | Too Many Requests |
| `500` `502` `503` `504` | Server Error — temporary, safe to retry with the same idempotency key |

---

## Transactional status codes

Returned in the `status` field of the response body on **payments, payouts, airtime, and bill** endpoints.

### Successful

| Code | Description |
|------|-------------|
| `20` | **Success** — transaction completed and funds moved |

### Pending

| Code | Description |
|------|-------------|
| `0` | **Created** — Tupay accepted the request but has not yet submitted it to the provider |
| `3` | **Submitted** — request forwarded to provider, awaiting response |
| `31` | **Processing** — provider is still processing the request |
| `100` `101` | **Retrying** — Tupay is automatically retrying with the provider |

### Failed

| Code | Description |
|------|-------------|
| `1` | **Cancelled** — you or the customer cancelled before completion |
| `2` | **Failed** — generic failure from the provider |
| `5` | **Invalid** — parameters accepted but provider rejected them (e.g. bad account number) |
| `6` | **Timeout** — provider did not respond within the configured window |
| `7` | **Insufficient funds** — your Tupay balance or the customer's balance is too low |
| `8` | **Reversed** — provider reversed the transaction after an earlier success |
| `9` | **Refunded** — you or Tupay initiated a refund |
| `10` | **Forbidden** — business rules prevent this transaction |
| `11` `120` | **Rejected** — provider explicitly rejected the transaction |
| `13` `15` `16` `24` | **Error** — unexpected provider or internal error; treat as failed and investigate |
| `17` | **Duplicate** — a transaction with the same idempotency key was already processed |
| `18` | **Expired** — transaction expired before completion |
| `19` | **Inactive** — target account or service is inactive |

## Handling statuses

<AccordionGroup>
  <Accordion title="Safe to retry">
    You can safely retry **with the same idempotency key** when:

    - HTTP status is `5xx`, or
    - Transactional `status` is `6` (Timeout) or an Error code (`13`, `15`, `16`, `24`).

    Tupay will deduplicate retries and ensure funds are not moved twice.
  </Accordion>

  <Accordion title="Do not retry automatically">
    Do **not** automatically retry when:

    - HTTP status is `4xx` (`400`, `401`, `403`, `409`), or
    - Transactional status is `1`, `2`, `5`, `7`, `10`, `11`, `17`, `18`, `19`, or `120`.

    In these cases, show the error to the user or correct the request before retrying.
  </Accordion>
</AccordionGroup>

When retrying after `5xx` or `429`, use **exponential backoff** (e.g. wait 1s, then 2s, then 4s) and respect the `Retry-After` header on 429. See [Rate limiting](/prerequisites/rate-limiting) for details.

## Error responses

When a request fails, the body may use one of these shapes:

**Simple error string:** many endpoints return a JSON object with an `error` field:

```json
{ "error": "Too many requests" }
```

**Structured response:** some endpoints include a `status` and a human-readable field such as `message` from third party providers:

```json
{ "status": 400, "message": "missing_params" }
```

**Target shape (for new and updated endpoints):** a consistent error object may include `type`, `code`, `message`, and `request_id` to support machine-readable handling and support lookup:

```json
{
  "error": {
    "type": "invalid_request_error",
    "code": "missing_params",
    "message": "Your request is invalid.",
    "request_id": "req_abc123"
  }
}
```

Use the HTTP status code to detect failure; parse `error` or `message` (and optional `status`/`code`) to show or log a reason. When contacting support, include the `X-Request-Id` response header if present (see [Request ID for support](#request-id-for-support) below).

### Request ID for support

Every API response includes an **`X-Request-Id`** header with a unique value for that request. If you contact Tupay support about an error or unexpected behaviour, include this value so we can look up the request in our logs.
